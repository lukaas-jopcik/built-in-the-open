# Evaluation — Skillcage

## Verdict: partial

## Depth: adequate — real (if self-contained) attack corpus and a genuine sandbox implementation were exercised and independently broken, but every target was synthetic (local JSON fixtures, no real tool/API integration) and the block-rate numbers came from a corpus that, by design, couldn't surface the resource-exhaustion class of bug that actually breaks it.

## Criteria
- [x] `python3 run.py examples/report.sky` exits 0 and `report.json` matches `examples/report.expected.json` — evidence: ran it, `diff report.json examples/report.expected.json` produced no output, exit code 0.
- [x] `python3 run.py examples/malicious.sky` blocks 100% of its 10 techniques, verified by `python3 -m unittest discover tests/` — evidence: `[SCOREBOARD] 10/10 attack vectors blocked, 0 escapes`, exit 0; `python3 -m unittest discover tests/` → `Ran 58 tests in 1.876s OK`.
- [x] `python3 benchmark.py` reports ≥90% of the 20-attack corpus blocked, 0 false positives, prints overhead % — evidence: `[BENCHMARK] attack corpus: 20/20 blocked (100.0%)`, `false positives: 0/5`, `overhead: safe_eval is +43.4% vs raw eval() (9.58µs vs 6.68µs/call, 100 iters)`.
- [x] `dashboard.html` via `python3 -m http.server` plays clean run then attack run with red BLOCKED banners and a final scoreboard, zero console errors — evidence: loaded it with Playwright/Chromium at 1280×800, `console/page errors: []`, screenshot shows the "10/10 attack vectors blocked, 0 escapes" banner and the parallel fan-out scene rendering correctly (`/tmp/dash_final.png`). Caveat below.
- [x] Zero third-party pip installs, no network calls beyond local `http.server` — evidence: `pip list` shows only pre-existing global packages (playwright, virtualenv, etc.) unrelated to this project; every script (`run.py`, `benchmark.py`, `fuzz_corpus.py`, `parallel_bench.py`) imports only stdlib (`ast`, `json`, `signal`, `concurrent.futures`, `time`).
- [✗] "Refuses every dangerous node... without crashing the interpreter" (Slice 2's own success bar, restated in the dashboard's "BLOCKED" promise) — **not met**. Found two one-line `.sky` expressions that crash the process with an uncaught Python traceback instead of producing a `BLOCKED` status:
  - `'a' * 999999999999999` → uncaught `MemoryError` inside `safe_eval.py:63`, `run.py` exits 1 with a raw stack trace.
  - `'+'.join(['1']*5000)` (i.e. `1+1+1+...+1`, 5000 terms) → uncaught `RecursionError` in `_eval_node`, same crash/exit-1 behavior.
  Neither is caught by `executor.py`'s exception handling (`run_sky` only catches `UnsafeExpressionError`, `StepTimeout`, `ToolError` — `MemoryError`/`RecursionError` propagate straight through). Neither is in the 20-item corpus or the 154-variant fuzz corpus (both target `Pow`/`LShift`-style operators per `attack_19`/`attack_20`; nobody tried `Mult`-based string repetition or a long flat `+` chain). Also `signal.alarm`-based timeout does not help here: the `MemoryError` fires in well under 2s, and the `RecursionError` is a stack-depth crash, not a wall-clock one.

## What broke / limitations
- **Real crash, not just a "leak"**: the PRD explicitly designed the sandbox to turn dangerous input into a caged `BLOCKED` step so "one caged attack step doesn't take down the rest of an independent DAG" (executor.py's own docstring). The two inputs above take down the *entire run* with a Python traceback and exit code 1 — worse than several of the things it successfully blocks, and exactly the failure mode Slice 2 was written to prevent.
- The attack/fuzz corpora both probe the AST *node-type* whitelist (Call, Attribute, Import, Pow, LShift, etc.) but never probe *resource exhaustion within allowed nodes* (Mult/BinOp chains). A 100% block rate against a corpus that doesn't include the attack that actually breaks it is a measurement gap, not a security guarantee — the corpus size (20 hand-written + 154 fuzzed) sounds broad but the fuzz generator (`fuzz_corpus.py`) recombines the *same* known gadget primitives (import/eval/exec/mro) rather than generating novel operator-abuse patterns, so it structurally can't find this class of bug.
- The dashboard's attack-run animation has a real (if minor) visual bug: BLOCKED banners never clear/fade, so by the 8th–10th attack node they stack on top of each other and on the node labels, producing an unreadable overlapping-text block for several seconds (see screenshot taken mid-run). It resolves once the scoreboard banner takes over, but a live demo audience would see the jumble first. The README itself flags that the dashboard was never visually verified in a real browser during the build — that admission checks out, and the flaw it would have caught is exactly this one.
- `run_sky_parallel`'s per-step timeout is enforced via `Future.result(timeout=...)`, but an over-time worker thread is only abandoned, not killed (Python has no safe thread-kill) — a thread stuck in a C-level allocation (e.g. the same string-multiplication case, run under `--parallel`) keeps consuming memory/CPU in the background even after the wave "times out" and moves on. Not tested end-to-end here, but it follows directly from the documented design and the crash found above.
- Everything unrelated to the sandbox boundary — DAG topological sort, cycle/duplicate-id detection, trace emission, the parallel speedup, the benchmark/fuzz harnesses themselves — worked exactly as documented, no complaints there.
- This is explicitly (and honestly) scoped as a language-level mitigation, not a security boundary — no seccomp/container/VM isolation, same-process, same-user. That's disclosed in the PRD/README, not a hidden gap. The gap that *is* hidden is that even within its own stated scope (AST whitelist should never let the interpreter crash), it doesn't hold.

## Founder translation
The core pitch — "our sandboxed tool-runner won't let an AI agent's tool call touch your filesystem, shell, or network" — mostly holds up: 20/20 hand-written attacks and 154/154 auto-generated variants (`os.system`, `eval`, reading `/etc/passwd`, socket connections, `__class__.__mro__` gadgets) were all blocked cleanly, for a ~43% CPU-time tax versus not sandboxing at all — cheap insurance. But a founder shipping this as-is would be exposed to a different, simpler problem: a single malformed or adversarial input expression (not even a "hack," just a long arithmetic string) crashes the whole process instead of being safely rejected. In production that's the difference between "the sandbox flagged step 7 and the pipeline kept going" and "your orchestration service returned a 500 and dumped a stack trace to the caller" — a reliability bug dressed up as a security demo. Bottom line: worth building on, not worth shipping around real user input yet; budget another pass specifically hardening the "allowed but unbounded" operators (string/collection repetition, expression depth) before trusting it with anything but hand-picked internal `.sky` files.

## Numbers
- 58/58 unit tests pass in 1.876s; 20/20 (100%) hand-written attack corpus blocked; 154/154 (100%) fuzzed gadget variants blocked; 0 false positives across 46 legitimate expressions.
- Sandbox overhead: +43.4% vs raw `eval()` (9.58µs vs 6.68µs per call, 100 iterations) — the measured "cost of the cage."
- Parallel fan-out: 300.9ms sequential → 63.8ms concurrent = 4.72x speedup on a 5-branch DAG.
- 2 previously-unknown crash inputs found in ~15 minutes of manual probing (`'a' * 999999999999999` → `MemoryError`; a 5000-term `+` chain → `RecursionError`), both bypassing all exception handling and killing the process with exit code 1 — 0 of these are covered by the project's own 174-case attack/fuzz corpus.
