# Evaluation — barebrowse

## Verdict: partial

## Depth: adequate — the snapshot/benchmark engineering is genuinely tested against 20 diverse live pages with a quantified, cross-checked comparison, but the headline "local model drives the agent" claim was never actually exercised in the shipped artifacts and fails when independently tested.

## Criteria

- [x] `snapshot <url>` non-empty, no traceback, 5 distinct real pages — evidence: ran against `Special:Random`, `Python_(programming_language)`, `example.com`, `news.ycombinator.com`, `python.org`; all exit 0, 11–3951 lines of structured `role[ref] "name"` output each.
- [x] `bench.py` ≥15 real pages, mean reduction ≥70% — evidence: `python3 barebrowse/bench.py --pages pages.txt --out bench_results.json` → 20/20 pages, **mean token reduction: 82.3%** (2,276,946 → 308,324 tokens). Cross-checked one page (Python_(programming_language)) against a real `tiktoken` `cl100k_base` encoder instead of the project's own regex-based estimator: 88.0% (project estimator) vs **82.0% (real BPE tokenizer)** — the custom estimator inflates the number by ~6pp because it counts every HTML punctuation character as a token, but the claim survives under a real tokenizer.
- [x] `Browser.goto/click/type` scripted 3+ step nav, automated, exit 0 — evidence: `tests/test_browser.py` passes (3/3) — but only via `python3 -m unittest tests.test_browser`. The PRD's literal instruction `python3 -m pytest tests/test_browser.py -q` **fails in the "stdlib only" clean venv** with `No module named pytest` (pytest is not stdlib). This is a test-plan documentation bug — the test file itself is pure `unittest` and works fine when invoked correctly; full suite is 33/33 passed once pytest is installed separately.
- [x] Agent completes task ≥8/10 runs, logged to `run_log.jsonl` — evidence: rule-based fallback, `python3 barebrowse/agent.py --task task.json --runs 10 --out run_log.jsonl` → **10/10 (100%)**, correctly extracts `1956-01-31` every run. Formally meets the criterion (PRD explicitly allows the rule-based fallback as the qualifying path).
- [~] "Autonomous agent" via local model (the PRD's actual wow-shot mechanism) — evidence: the shipped `run_log.jsonl`/`report.html` in the repo were generated entirely with `"policy": "rule"`, never Ollama. I installed Ollama + `llama3.2` (3B, Q4_K_M) myself and re-ran with `--policy ollama`: **1/10 (10%) nominal success**, and that single "success" is a false positive — the model said `done` on the Wikipedia Main Page itself (never navigated anywhere) with `extracted: "<This page contains a list of languages>"`, which doesn't remotely match the birth-date regex or the task. Root cause: `_parse_llm_action()` in `agent.py` hard-codes `success: True` for any `ACTION: done` line the model emits, with **no check that `extracted` actually satisfies `extract_regex`** (the rule-based policy does this check; the LLM path doesn't). The other 9/10 runs crashed on hallucinated/malformed refs (e.g. `click [e645]` when no such ref existed, `type "Shqip" ...`). True verified success rate for the real local-model path: **0/10**.
- [x] `report.html` standalone, brand-styled, no server — evidence: regenerated via `render_report.py`, 201KB single file, dark theme with custom CSS variables (`--periwinkle: #8c9eff`, Fraunces/Inter fonts), per-page token bar chart, embedded `<script>` passes `node --check`. Minor caveat: it loads Google Fonts from `fonts.googleapis.com` via CDN `<link>` — not literally 100% offline/standalone (falls back to system serif/sans-serif if no network, doesn't break, just looks plainer). Could not visually screenshot in this sandbox (no headless browser installed) — verified structurally instead of visually.
- [x] Core snapshot/browser/bench stdlib-only — evidence: created a real clean venv (`python3 -m venv` + `ensurepip`, zero packages via `pip list`), ran `snapshot`, `bench.py`, and the browser test successfully with no third-party installs.

## What broke / limitations

- **The core "local model" claim in the PRD title/goal is not substantiated.** The shipped demo artifacts (`report.html`, `run_log.jsonl` in the repo) were built from the deterministic rule-based fallback, not an actual model. When I stood up the real thing (Ollama + llama3.2, exactly per the PRD's own optional instructions), it worked essentially 0/10 times on the one task the whole project is built around, with a scoring bug masking that further.
- **Scoring bug**: `agent.py`'s `_parse_llm_action` trusts the model's self-reported "done"/success unconditionally; it never validates against `extract_regex` the way the rule-based policy does. Any future run of the ollama path will over-report success.
- **The "rule-based fallback" is not a general agent** — it's a generic 4-state template (find textbox → submit → find link containing a substring → regex-extract) that works for single-search-then-extract tasks shaped exactly like `task.json`. It has no error recovery, no re-planning, and would not generalize to multi-hop tasks, disambiguation pages, or sites without a simple search box.
- **Token counting is a custom regex approximation**, not a real tokenizer (confirmed ~6 percentage-point inflation vs. `tiktoken` on a spot check) — the ratio is directionally right but the exact "82.3%" number is somewhat estimator-dependent.
- **Small-model failure mode is real and likely persists**: llama3.2:3B repeatedly hallucinated refs not present in the snapshot it was just shown, even though the snapshot is short and explicit about valid `[ref]` tokens — larger local models might do better, but this wasn't tested (no time/hardware budget in this review), so it's an open question, not a resolved one.
- **No JS/SPA support** (explicitly out of scope, correctly disclosed) — will silently degrade to whatever server-rendered HTML shell a modern JS-heavy site ships, likely mostly empty snapshots on such sites (not tested here since out of scope by design).
- **No auth-gated flows, no POST-form completion** — `Browser.click` explicitly raises on POST forms rather than silently downgrading (good, deliberate safety choice, verified via `test_post_form_is_refused_not_silently_downgraded`).
- Fetch/error handling is solid: verified graceful handling of 404s, DNS failures, timeouts, and malformed/unclosed HTML tags — no raw tracebacks in any case tried.

## Founder translation

The part of this that actually works reliably is a webpage-shrinker: it takes a real webpage and strips it down to ~18–20% of its size before any AI model ever reads it, which is a genuine, measurable cost and context-window saving if you're paying per token for an LLM to read web pages. What does **not** work yet is the more exciting claim — a small model "browsing the web on its own" using this shrunk-down version. When tested for real with an actual local model, it got lost and picked broken links on 9 out of 10 tries, and the demo video/report that exists today was quietly generated by a hardcoded script standing in for the AI, not the AI itself. So: real savings on the token bill, but the "autonomous local agent" pitch isn't demo-safe yet — showing this to a customer or investor as-is risks someone asking to see it live with a real model and it failing in front of them.

## Numbers

- Mean token reduction across 20 real pages: **82.3%** (own estimator) / **82.0%** (independently cross-checked with `tiktoken cl100k_base` on the largest page, vs. the project's own reported 88.0% for that same page).
- Rule-based fallback agent success rate: **10/10 (100%)**.
- Real local-model (llama3.2:3B via Ollama) agent success rate: **1/10 nominal, 0/10 verified** (the one "success" is a false positive from an unvalidated scoring path).
